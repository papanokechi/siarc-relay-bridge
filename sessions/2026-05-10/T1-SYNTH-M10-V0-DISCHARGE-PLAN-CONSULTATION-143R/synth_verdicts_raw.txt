### Q4 -- S1/S2 proof-shape strategy

**(a) Axiom reformulation vs Mathlib-native proof-shape:**

The triage's framing -- "split `frobenius_double_root_at_apparent_singularity` into a version that takes only `(ha_root, ha_simple)` and absorbs the derivative-shape hypothesis internally" -- is the **right strategic direction**, but the precise refactor depends on what `frobenius_double_root_at_apparent_singularity` actually claims as a theorem (which I cannot fully resolve without the Lean source at HEAD; APPENDIX C only shows the call site, not the axiom statement).

Two refactor patterns to consider, in order of preference:

**Pattern α (preferred): Drop the `h_exact` hypothesis from the axiom.**
If the axiom currently reads (schematically):
```
axiom frobenius_double_root_at_apparent_singularity :
  ∀ (a c : ℂ → ℂ) (s : ℂ),
    is_root_of_indicial a s →
    is_simple_root a s →
    (∀ x, HasDerivAt (fun x => a x * deriv id x) (a x) x) →  -- h_exact (the artefact)
    apparent_singularity_at s
```
then the `h_exact` clause is structurally redundant when `id` is the identity (`deriv id x = 1` reduces the goal to `HasDerivAt a (a x) x` which is false in general -- so the axiom as currently stated cannot be applied except in degenerate cases anyway). The refactor is to **remove the `h_exact` parameter from the axiom signature**, leaving:
```
axiom frobenius_double_root_at_apparent_singularity :
  ∀ (a c : ℂ → ℂ) (s : ℂ),
    is_root_of_indicial a s →
    is_simple_root a s →
    apparent_singularity_at s
```
This closes S1 and S2 by **deletion** of the `(by sorry)` argument at the call sites (L118 and L120), not by discharge. Phase C effort: ~5 LOC delta in `Thm66_ApparentSingularity.lean`. **This is structural revision and triggers the BAND ceiling per sec 3.2.**

**Pattern β (fallback if axiom-content requires `h_exact` for downstream proofs):** Replace `h_exact` with a math-content-equivalent hypothesis that is dischargeable for the actual `a_coeff_c`. For example, if the axiom genuinely needs to know that `a` is non-degenerate at `s`, the replacement hypothesis should be `a s ≠ 0 ∧ deriv a s ≠ 0` (already available as `a_deriv_s1_ne_zero` etc. -- which suggests the axiom is *already* receiving the right hypothesis through other parameters and `h_exact` is pure redundancy). Pattern β collapses to Pattern α in that case.

**Mathlib-native proof-shape avoiding axiom changes:** NOT FEASIBLE. The current `h_exact` shape is mathematically false for `a = a_coeff_c` (per APPENDIX A sec 3 S1 analysis), so no Mathlib `Deriv.*` lemma can close it. Any attempt to discharge as currently stated is a fabricated-math risk.

**(b) Discharge ordering -- before vs after Phase C.3:**

CONCUR with triage's recommendation: **after C.3**. Reasoning:
- Pattern α (axiom-signature edit) touches `Thm66_ApparentSingularity.lean` only; build-graph-localized.
- WallisFamily build-fix (B1-B5) is independent of Thm66 axioms (per APPENDIX D dependency_map: no edge from WallisFamily.lean to Thm66_ApparentSingularity.lean).
- Greening WallisFamily (C.1) + commit (C.3) hardens the substrate for the next agent fire (slot 142-class for the axiom refactor).
- Doing the axiom refactor first risks blast-radius coupling with WallisFamily build-fix (operator changing two files in iteration loop = harder to localize regressions).

**Recommended sequencing:**
1. C.1: green WallisFamily (B1-B5 discharge in iter-14 / iter-15)
2. C.3: commit lean/ working tree
3. C.3+: clean-clone reproducibility check (per slot 144 Change 3)

See amendment C-143-4.

EXCERPT-D1 -- V_quad PCF family + recurrence + Newton polygon
----------------------------------------------------------------

V_quad PCF family is the degree-2 PCF defined by:

  recurrence:   p_{n+1} = a_n p_n + b_n p_{n-1}
  with:         a_n = 1
                b_n = 3 n^2 + n + 1

(Recurrence stored in pcf-research/channel/cc_pipeline_g_2026-05-01/
results_vquad.json with the canonical leading-first PCF convention
[a_2, a_1, a_0] for a-coeffs and [b_1, b_0] for b-coeffs.)

Newton polygon at z = 0 of the linear ODE for the generating function
f(z) = sum_n Q_n z^n:

  slope:                    1/2
  edge:                     (0, 0) -- (1, 2)
  multiplicity:             2
  characteristic polynomial: 1 - (3/4) c^2
  Gevrey class in z:        2
  Gevrey class in u = sqrt(z): 1

So the V_quad generating function f(z) has Gevrey-1 formal series in
the regularising coordinate u = sqrt(z), characteristic exponent
c = +/- 2/sqrt(3), recovering the literature constant
xi_0 = 2/sqrt(3) (numerical agreement to 200 digits).

CRITICAL LABELING-CONVENTION NOTE (synth must disambiguate):

The V_quad reduction script at
pcf-research/channel/cc_pipeline_2026-05-01/vquad_p3d6_recovery.py

  "source": "p12_journal_main.tex sec:vquad subsec Stokes data,

This is the CLASSICAL-ODE 4-tuple convention (alpha, beta, gamma,
delta of Okamoto eq. (0.1)) per the §3.5.1 Remark
[rem:alpha-beta-tuples].

HOWEVER, CT v1.3 §3.5.1 (105 deposit, bridge SHA 0427c0a) declares
the *same* 4-tuple (1/6, 0, 0, -1/2) at the V_quad parameter point
under the HAMILTONIAN-PARAMETER convention (alpha_inf, alpha_0,
beta_inf, beta_0) per the trivial relabel (3.5.1a)-(3.5.1d).

These two conventions are DIFFERENT objects. Under Okamoto eq. (0.1):

  alpha_ODE   = -4 eta_inf theta_inf
  beta_ODE    = 4 eta_0 (theta_0 + 1)
  gamma_ODE   = 4 eta_inf^2
  delta_ODE   = -4 eta_0^2

If (alpha, beta, gamma, delta) = (1/6, 0, 0, -1/2):
  gamma = 4 eta_inf^2 = 0  =>  eta_inf = 0
  But Okamoto §1 explicitly requires eta_Delta != 0 (assumption
  gamma delta != 0 throughout the paper).

Therefore one of the following is true:
  (A) The vquad_p3d6_recovery.py LIT dict mislabels the tuple; the
      tuple is actually (alpha_inf, alpha_0, beta_inf, beta_0) per
      §3.5.1, NOT (alpha_ODE, beta_ODE, gamma_ODE, delta_ODE).
  (B) The vquad_p3d6_recovery.py uses a non-standard P_III
      parametrisation outside Okamoto §1's normal form.
  (C) Some other resolution.

Synth: please weigh (A) vs (B) vs other in QD-5 below.
